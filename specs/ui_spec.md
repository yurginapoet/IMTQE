# MTQE — Спецификация интерфейса и интерпретации оценок

> Рабочий документ. По нему реализуем `src/app/`.

---

## 1. Общая архитектура приложения

### 1.1 Стек

```
Backend:   FastAPI + uvicorn
           src/app/api.py          — REST-эндпоинты
           src/app/server.py       — точка запуска, startup-хук загрузки моделей

Frontend:  Jinja2 шаблон + Vanilla JS (без фреймворков)
           src/app/templates/index.html
           src/app/static/app.js
           src/app/static/style.css
```

### 1.2 Загрузка моделей — один раз, навсегда

```python
# server.py / api.py
@app.on_event("startup")
async def load_models():
    app.state.predictor = Predictor(...)   # LaBSE + ruGPT-3 + XLM-R + NGBoost
```

**Требование:** модели загружаются при старте процесса uvicorn.
Перезагрузка страницы браузером НЕ вызывает повторную загрузку.
Модели живут в `app.state.predictor` на весь жизненный цикл процесса.

Эндпоинт `GET /api/status` возвращает `{"ready": true/false}` —
фронтенд проверяет его при загрузке страницы и блокирует ввод
пока модели не готовы (показывает спиннер "Загрузка моделей...").

## 3. Таблица сегментов (CAT-режим)

### 3.1 Строка сегмента

Каждая строка содержит:

| Элемент | Описание |
|---|---|
| `#N` | Номер сегмента, кликабельный — активирует панель деталей |
| `textarea` SRC | Оригинал EN, редактируемый, авто-resize по высоте |
| `textarea` MT | Перевод RU, редактируемый |
| Score-badge | Цветной бейдж с числом; если оценка ещё не готова — спиннер |
| Кнопка удаления | Появляется при hover на строке, недоступна если строка единственная |

### 3.2 Отображение перевода с подсветкой

**Когда оценка получена**, `textarea` переключается в режим отображения:
текст рендерится как `div` с inline HTML-подсветкой (`highlighted_mt_html`).
При клике на `div` он снова становится редактируемым `textarea`.

Подсветка слов:
- `BAD-major` → красный фон `#FFCDD2`, жирная красная нижняя граница
- `BAD-minor` → жёлтый фон `#FFF9C4`, жёлтая нижняя граница
- `OK` → без подсветки

**Tooltip при наведении на подсвеченное слово:**
```
Тип ошибки:    Fluency/LexicalChoice
Описание:      Неудачный выбор слова — нетипичная для русского текста лексика
Severity:      BAD-minor
Уверенность:   73%
```

Tooltip реализуется как абсолютно позиционированный `div`,
появляется через 200ms задержки, исчезает при уходе мыши.

### 3.3 Score-badge

```
≥ 0.80  →  зелёный    ●  0.82   "Хороший перевод"
0.60–0.79 → жёлтый   ●  0.71   "Требует внимания"
0.40–0.59 → оранжевый ●  0.45  "Есть ошибки"
< 0.40  →  красный    ●  0.28  "Плохой перевод"
```

При hover на badge — краткий tooltip: "MQM score: 0.48 · CI₉₅: [0.38–0.54]"

### 3.4 Кнопки управления строками

- **[+ Добавить строку]** — добавляет пустую строку под последней
- **[× удалить]** — удаляет строку (hover на строке); недоступно если 1 строка
- Строки не перетаскиваются (не нужно в MVP)

## 4. Автоматический запуск оценки

### 4.1 Триггер

Оценка запускается автоматически при:
- Потере фокуса (`blur`) из поля MT или SRC данного сегмента
- Если с момента последнего изменения прошло > 1.5 сек (debounce)

**НЕ** запускается:
- При каждом нажатии клавиши (слишком часто)
- При пустых обоих полях

### 4.2 Последовательность, не параллельность

Если в таблице N сегментов и все требуют оценки:
- Запросы отправляются **по одному**, последовательно
- Пока один сегмент оценивается — его badge показывает спиннер
- Следующий запрос отправляется только после ответа на предыдущий
- Это даёт интерактивность: результаты появляются по мере готовности

Реализация через async queue в JS:
```javascript
const evalQueue = new EvalQueue();
// при изменении сегмента:
evalQueue.enqueue(segmentId);
// queue обрабатывает по одному запросу
```

### 4.3 Состояния сегмента

```
idle      → не оценивался (серый badge "—")
pending   → в очереди (серый badge со спиннером)
loading   → запрос отправлен (анимированный спиннер)
done      → оценка получена (цветной badge)
error     → сервер вернул ошибку (красный badge "ERR")
dirty     → сегмент изменён после последней оценки (badge с иконкой ✏)
```

### 5.3 Секция "Итоговая оценка" — интерпретация

Правила перевода числа в текст (отображается рядом с score):

| Score | Метка | Цвет | Пояснение для пользователя |
|---|---|---|---|
| 0.80–1.00 | **Хороший** | зелёный | Перевод точен и грамотен. Публикация без правок |
| 0.65–0.79 | **Приемлемый** | жёлто-зелёный | Незначительные недочёты. Рекомендуется лёгкая правка |
| 0.50–0.64 | **Требует правки** | оранжевый | Заметные ошибки. Необходима редактура |
| 0.35–0.49 | **Плохой** | красный | Серьёзные искажения смысла или грамматики |
| 0.00–0.34 | **Очень плохой** | тёмно-красный | Требуется полный перевод заново |

**Неопределённость (uncertainty):**

| Var[Beta] | Метка | Пояснение |
|---|---|---|
| < 0.01 | низкая | Признаки согласованы, оценка надёжна |
| 0.01–0.03 | умеренная | Есть противоречия между признаками |
| > 0.03 | высокая | Признаки сильно расходятся, проверьте перевод вручную |

### 5.4 Секция "Вклад по категориям" — интерпретация SHAP

Каждый признак из 22 отображается не напрямую, а через агрегацию
по MQM-категориям (из `explanation` поля результата).

**Правила отображения вклада:**

```
shap_value > +0.05  →  "↑ улучшает оценку"    зелёный
shap_value ∈ [-0.05, +0.05]  →  "↑ норма"     серый
shap_value < -0.05  →  "↓ снижает оценку"     красный
```

Прогресс-бар: длина пропорциональна |shap_value|, цвет по знаку.

**Человекочитаемые расшифровки по категории + знаку:**

```
Accuracy (Точность):
  отриц. → "Смысл оригинала передан неточно. Проверьте пропуски или искажения."
  полож. → "Смысловая точность в норме."

Fluency (Грамотность):
  отриц. → "Текст перевода звучит неестественно. Возможны грамматические ошибки
             или кальки с английского."
  полож. → "Текст грамотен и естественно звучит по-русски."

Terminology (Терминология):
  отриц. → "Специальные термины или имена собственные переведены неточно
             или не сохранены."
  полож. → "Терминология передана корректно."

Locale (Локаль):
  отриц. → "Обнаружены проблемы с форматированием: числа, даты, кавычки
             или символы валют не адаптированы для русского текста."
  полож. → "Форматирование соответствует русской локали."

Style (Стиль):
  отриц. → "Стиль или регистр перевода не соответствуют оригиналу
             (например, слишком формально или разговорно)."
  полож. → "Стиль и регистр соответствуют оригиналу."
```

### 5.5 Секция "Найденные ошибки"

Список спанов из `errors`. Для каждого:

```
[severity-dot]  "текст спана"   Краткий тип · Полный тип · Уверенность%
```

Severity dot:
- `BAD-major` → красный кружок ●
- `BAD-minor` → жёлтый кружок ●

Краткий тип (для пользователя):

| error_type | Краткий label |
|---|---|
| Accuracy/Mistranslation | Искажение смысла |
| Accuracy/Omission | Пропуск фрагмента |
| Accuracy/Addition | Лишний фрагмент |
| Accuracy/Untranslated | Не переведено |
| Accuracy/Hallucination | Выдуманный смысл |
| Fluency/Morphology | Морфологическая ошибка |
| Fluency/Agreement | Ошибка согласования |
| Fluency/Spelling | Орфографическая ошибка |
| Fluency/LexicalChoice | Неудачная лексика |
| Terminology/WrongTerm | Неверный термин |
| Terminology/Inconsistency | Разные переводы одного слова |
| Locale/NumberFormat | Формат числа |
| Locale/DateFormat | Формат даты |
| Locale/Quotes | Неверные кавычки |
| Locale/Currency | Обозначение валюты |
| Style/Register | Несоответствие регистра |

При клике на строку ошибки — соответствующее слово в MT-ячейке
подсвечивается пульсирующей рамкой (CSS animation).

## 6. Ручная разметка ошибок (фидбек для дообучения)

### 6.1 Кнопка "Отметить ошибку вручную"

Находится в панели деталей внизу.
Доступна только для сегментов с готовой оценкой.

### 6.2 Флоу разметки

1. Пользователь нажимает "Отметить ошибку вручную"
2. MT-поле активного сегмента переходит в режим выделения:
   курсор меняется, появляется подсказка "Выделите ошибочный фрагмент"
3. Пользователь выделяет текст мышью
4. Появляется mini-форма рядом с выделением:
   ```
   Тип ошибки:  [dropdown MQM типов]
   Severity:    ○ BAD-minor  ○ BAD-major
   [Сохранить]  [Отмена]
   ```
5. При нажатии "Сохранить":
   - `POST /api/feedback` с `{seg_id, src, mt, start_char, end_char, error_type, severity, features}`
   - features берутся из кэша последней оценки этого сегмента
   - Разметка добавляется в `data/feedback/feedback.jsonl`

### 6.3 Визуализация пользовательских меток

После сохранения — в панели деталей появляется секция
"Ваши пометки" с синим кружком ● (отличается от автоматических).

## 7. Эндпоинты API

```
GET  /api/status
     → {"ready": bool, "models_loaded_at": timestamp}

POST /api/evaluate
     body: {"src": str, "mt": str}
     → SentenceUIResult (JSON)

POST /api/evaluate_batch
     body: {"pairs": [{"src": str, "mt": str}, ...]}
     → [SentenceUIResult, ...]

POST /api/feedback
     body: {
       "src": str,
       "mt": str,
       "start_char": int,
       "end_char": int,
       "error_type": str,     # из MQM_ERROR_TYPES
       "severity": str,       # "BAD-minor" | "BAD-major"
       "features": dict,      # 22 признака из последней оценки (кэш фронтенда)
       "word_logprobs": list  # из последней оценки
     }
     → {"saved": bool, "feedback_id": str}

GET  /
     → HTML интерфейс (Jinja2 шаблон)
```

## 8. Кэш признаков на фронтенде (для фидбека)

После каждой оценки фронтенд сохраняет в памяти (JS Map):

```javascript
segmentCache.set(segmentId, {
  features: result.debug.features,   // 22 признака
  word_logprobs: result.debug.word_logprobs,
  evaluated_at: Date.now(),
  src: srcText,
  mt: mtText,
});
```

При отправке фидбека фронтенд берёт features из кэша —
сервер **не пересчитывает тяжёлые признаки**.

`/api/evaluate` должен возвращать поле `debug: {features: {...}, word_logprobs: [...]}`
(уже есть в `SentenceUIResult`).

## 9. Дообучение офлайн (offline fine-tuning)

### 9.1 Формат файла фидбека

`data/feedback/feedback.jsonl` — каждая строка:

```json
{
  "id": "uuid4",
  "timestamp": "2024-01-15T10:23:00",
  "src": "The bank approved the loan.",
  "mt": "bank одобрил кредит.",
  "start_char": 0,
  "end_char": 4,
  "error_type": "Accuracy/Untranslated",
  "severity": "BAD-major",
  "features": {"cosine_similarity": 0.87, "length_ratio": 0.95, ...},
  "word_logprobs": [-3.1, -2.8, -9.2, -3.4, -4.1]
}
```

### 9.2 Скрипт дообучения

`scripts/finetune_from_feedback.py`

**Принцип: тяжёлые признаки не пересчитываются.**

```
Входные данные:
  data/feedback/feedback.jsonl   ← features уже посчитаны

Что дообучается:
  A) NGBoost/XGBoost — incremental fitting на новых примерах
     Используем features из feedback напрямую.
     XGBoost: xgb.train(..., xgb_model=existing_model)  # warm start
     NGBoost: не поддерживает incremental — переобучаем на (old_train ∪ new_feedback)
              но old_train features уже в sentence_da_features.parquet — читаем оттуда.

  B) XLM-RoBERTa span model — дообучение на новых размеченных спанах
     Конвертируем char-level разметку → word-level метки через char offsets.
     Дообучаем на малом числе эпох (1-2) с низким lr (5e-6) чтобы не забыть старое.
     Используем только feedback с word_logprobs (word-level контекст уже есть).

Что НЕ пересчитывается:
  LaBSE embeddings, ruGPT-3 logprobs — берутся из features в feedback.jsonl.
```

**Команда:**
```bash
python scripts/finetune_from_feedback.py \
  --feedback data/feedback/feedback.jsonl \
  --model ngboost                          \  # или xgboost
  --min-samples 10                         \  # минимум примеров для запуска
  --output models/
```

### 9.3 Когда запускать

Рекомендуемая стратегия:
- Запускать вручную когда накопилось ≥ 10 новых записей в feedback.jsonl
- Или по расписанию (cron) ночью
- После дообучения сервер перезапускается (uvicorn --reload или systemctl restart)

**Горячая перезагрузка модели** (без перезапуска сервера):
```
POST /api/reload_models
     → {"reloaded": true}
     Перезагружает pkl/model файлы без перезагрузки LaBSE и ruGPT-3.
     Только sentence model и span model.
```

### 10.2 Интеграционный тест (с заглушкой Predictor)

```python
# tests/test_api.py
from fastapi.testclient import TestClient
from unittest.mock import MagicMock, patch

def test_evaluate_endpoint():
    # Мокаем Predictor чтобы не загружать 3.6GB моделей в CI
    with patch("src.app.api.get_predictor") as mock:
```

## 12. Ограничения MVP

- Только sentence-режим (не paragraph) в первой версии UI
- Tooltip только при hover, не при тапе (мобильные не поддерживаются)
- Фидбек сохраняется только локально в jsonl, не в БД
- Дообучение запускается вручную, не автоматически
- Максимум 50 сегментов в таблице (ограничение производительности CPU)
